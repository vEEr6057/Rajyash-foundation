# Food Porter — Production Cutover & Ownership Handover

*For the Rajyash Foundation co-founders · July 2026*
*Companion to [FOOD-PORTER-END-GOAL.md](FOOD-PORTER-END-GOAL.md) (vision & costs). This
document covers the practical side: what exists today, whose accounts it runs on, and the
exact steps that make everything the foundation's property.*

---

## 1. The short version

The Food Porter platform is built, tested, and ready. Today every service it runs on is
connected to the **developer's personal accounts** — that was right for building, wrong
for operating. Before launch on rajyashfoundation.com, ownership moves to the foundation:
every account on a foundation email, the developer staying on as an invited operator.
The result: if the developer disappears tomorrow, the foundation loses nothing.

## 2. Current state

- **The platform**: complete and audited — donor posting, pickup dispatch, live map
  tracking, delivery runs, admin dashboard, reports, online donations (Razorpay, test
  mode), three languages, installable on any phone.
- **The current rajyashfoundation.com website is broken** — it has been returning a
  server error (HTTP 500) since roughly April 2026 (WordPress site on a shared host).
  Whatever content matters from it is recoverable from the Internet Archive.
- **The domain** is registered at GoDaddy (that account is presumably the foundation's —
  needs confirming).
- **No email runs on the domain** (checked: no MX records) — so re-pointing it carries
  zero email risk.

## 4. What we need from the foundation

1. **One email address to own everything** — recommend a dedicated
   `foodporter@rajyashgroup.com` (or reuse `rajyashfoundation@rajyashgroup.com`).
   Every account above gets created on / transferred to it. Someone at the foundation
   must be able to receive its mail (verification links, alerts).
2. **A password manager** (free tier of Bitwarden is fine) holding: every account
   password, the two-factor recovery codes, the backup encryption passphrase, and the
   web-push keys. Two people at the foundation should have access — this vault IS the
   ownership.
3. **A phone number for two-factor authentication** on the critical accounts
   (Cloudflare, GitHub, Razorpay, Supabase).
4. **GoDaddy access** — confirm the foundation controls the GoDaddy account holding
   rajyashfoundation.com; we need one nameserver change made there (or delegate access
   to make it ourselves).
5. **Razorpay** — confirm the in-progress NGO KYC is under the foundation's PAN/bank
   account. If it is not, restart it under the foundation's — donations must settle
   directly to the foundation's bank account, never through an intermediary.
6. **The content decision** — the old site had pages for the foundation's other five
   programs (education, plantation, animal rescue, Anand Mela, other activities).
   Option A: we port them into the new site as static pages. Option B: launch
   food-rescue-focused and add them later. (Old content is recoverable either way.)
7. **~2 hours of a co-founder's time** during handover week — account creations need
   the email owner present (verification links, 2FA setup).

## 5. The handover plan, step by step

Order matters: accounts first, DNS switch last. Total effort ≈ 2–3 working days
(developer), spread over about a week (waiting on verifications). The app keeps working
throughout — users notice nothing until the new domain goes live.

### Step 1 — Foundation identity (co-founder + developer, ~1 hour)
Create the email (or confirm access), set up the password manager, enable 2FA as each
account below is created.

### Step 2 — GitHub (developer, ~1 hour)
Create a free GitHub organization for the foundation → transfer the repository into it
(GitHub preserves redirects; nothing breaks) → re-add the deployment secrets (they do
not transfer — by design) → re-run the checks to confirm deploys still work.

### Step 3 — Cloudflare (developer, ~half day)
Create the foundation Cloudflare account → add the developer as a member → deploy the
Worker there → move the deployment credentials in GitHub to point at the new account →
add rajyashfoundation.com as a zone (this produces the two nameservers for Step 7).

### Step 4 — Supabase (developer, ~1 hour)
Create a foundation Supabase organization → use Supabase's built-in project transfer to
move the database project into it → developer stays as member. Zero data movement — the
project moves with all data, storage, and settings intact.

### Step 5 — Clerk production instance (developer, ~half day)
Clerk requires a custom domain for production — we now have one. Create the production
instance under a foundation-owned Clerk workspace, configured for rajyashfoundation.com.
DNS records for it are added in the Cloudflare zone (Step 3). New live keys go into the
deployment secrets. *(The dev instance stays on the developer side for testing — by
design, see the environment-split practice.)*

### Step 6 — Resend + Inngest + monitoring (developer, ~1 hour)
Fresh foundation accounts; verify the domain with Resend (enables
`noreply@rajyashfoundation.com` as sender — this is what turns email from "sandbox, owner-only"
into real delivery); swap the API keys in deployment secrets. Create UptimeRobot on the
foundation email at the same time. (Sentry is optional — the built-in client-error logging
already covers the basics; add Sentry later only if wanted.)

### Step 7 — DNS cutover (the launch moment, ~15 min + up to 24h propagation)
At GoDaddy: switch the domain's nameservers to the two Cloudflare gave us in Step 3.
The new site goes live on rajyashfoundation.com as propagation completes. The old
hosting can then be cancelled (nothing else uses it — confirm with whoever pays for it).

### Step 8 — Rotate everything + clean slate (developer, ~1 hour)
Re-issue every API key and secret that existed before handover; store the new set in the
foundation password manager. From this moment no credential from the build phase works.
Since the platform has no real users yet (only the developer and a few testers), all test
pickups/users/donations are purged so the foundation launches with a clean database.
Final smoke test (sign-in, post a pickup, dispatch it through pickup → delivery, test donation)
and the launch checklist in the operations runbook.

### Razorpay (parallel track, not blocking)
Donations run in test mode until the foundation's KYC clears; the moment it does, live
keys drop in and the webhook URL is set to the new domain. Everything else launches
without waiting for this.

## 6. What the foundation owns after handover

- The domain, the website, the application code, and every user and rescue record.
- Every account it runs on, with the developer as a removable invited operator.
- Nightly encrypted database backups (90 days of history) under its own GitHub org.
- The password-manager vault — the master key to all of it.
- Zero monthly cost at launch; paid stages are the foundation's choice, laid out with
  prices in [FOOD-PORTER-END-GOAL.md](FOOD-PORTER-END-GOAL.md).

## 7. Developer attribution (part of this agreement)

The platform was designed and developed end-to-end by **Veer Shah** (2026). The handover
transfers *operation and ownership of the running service* to the foundation; it does not
erase authorship. The foundation and developer agree:

1. **Code license** — the source code is licensed to the foundation under the MIT license
   (see `LICENSE` in the repository), which grants the foundation free, perpetual,
   unrestricted use while preserving the author's copyright notice in the codebase. The
   `AUTHORS.md` file records authorship and remains in the repository.
2. **Acknowledgment** — the foundation acknowledges Veer Shah as the developer of the
   Food Porter platform, and this signed document serves as written confirmation of it.
3. **Portfolio rights** — the developer may publicly reference this project (name,
   screenshots, architecture write-ups, aggregate impact numbers) in his CV, portfolio,
   and professional profiles, indefinitely and regardless of future involvement.
   (User personal data is never part of this — only the product itself.)
4. **Public credit** — the foundation will credit the developer in the launch
   announcement and, on request, provide a reference letter / LinkedIn recommendation
   covering the engagement.
5. **Repository visibility** — preference: the repository remains public under the
   foundation's GitHub organization (it contains no secrets — they live in CI/Worker
   secret stores), which keeps the developer's contribution history publicly verifiable.
   If the foundation prefers it private, the developer retains a personal read-only fork
   for portfolio evidence.

Nothing above limits the foundation's ownership or ability to operate, modify, or hire
others to work on the platform.
