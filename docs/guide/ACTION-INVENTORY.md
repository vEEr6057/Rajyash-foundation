# Action Inventory — Rajyash Food Porter

Exhaustive, code-verified inventory of every action each persona can perform, for
screenshot-capture / user-handbook purposes. Generated by reading every server action
under `src/features/**/actions/` and every calling component, on `main`
(dispatch-model-v2 + form-standardization merged). Read-only audit — nothing in the
app was changed to produce this.

**Conventions used below:**
- **FormSheet** = the app's shared slide-over sheet (`src/components/forms` `FormSheet`)
  that hosts a form in place (replaces what used to be standalone `/…/new` pages —
  those routes now just `redirect()` to the list page where the sheet lives).
- **ConfirmDialog** = the shared confirm/cancel modal (`src/components/ui/confirm-dialog`).
- Route column shows the page the control lives on; sheets/dialogs have no route of
  their own (they open in place).

## Gaps & leftovers found (read this first)

1. **NotificationBell / NotificationFeed — built but not wired into any page.**
   `src/features/notifications/components/NotificationBell.tsx` is a fully working bell
   icon + unread badge + dropdown feed, backed by a real hook (`useNotifications`) and
   real server actions (`listNotifications`, `getUnreadCount`, `markNotificationRead`,
   `markAllNotificationsRead`). It is exported from `src/features/notifications/index.ts`
   but **never imported anywhere** — not in `AuthedHeader.tsx` (portal/admin header),
   not in `PublicHeader.tsx`, not in any layout. Only `PushOptIn` (the enable-push-toast
   card) from the same feature is actually rendered (on `/portal/dashboard`). Net effect:
   there is currently **no in-app way to see notifications** other than the push
   notifications themselves — the in-app bell/list has no capture-able entry point
   because it isn't on screen anywhere.
2. **`assignDriver` server action has no UI caller.** `src/features/runs/actions/runActions.ts`
   exports `assignDriver(runId, driverId)` (a dedicated "assign/reassign a driver to an
   existing run" action), but no component imports it — only `runActions.test.ts` calls
   it. The only way to set a run's driver in the UI is at **creation time**, via the
   driver `FormSelect` in `BuildRunForm` (opened from `NewRunSheet`). There is no way to
   assign/change a driver on an already-created run from the admin run detail page
   (`/admin/runs/[id]`) — `RunStatusControls` only offers status transitions + delete.
3. **`editRun` server action has no UI caller.** Same file exports `editRun(runId, input)`
   for changing an existing run's slot/date/driver — also only exercised by
   `runActions.test.ts`. No edit form exists on `/admin/runs/[id]`.
4. **Old-model leftover / likely bug: admin's manual "Assign" dialog on `/admin/pickups`
   still targets *volunteers*, not drivers.** Under dispatch-model-v2 the collector role
   is the **driver** — `claimPickup` and `advancePickup` both `requireRole(["driver"])`.
   But the admin-only manual-assign path (`PickupsTable.tsx` → `AssignDialog` →
   `assignPickup()` in `src/features/admin/actions/adminActions.ts`) sources its picker
   from `profilesRepo.listAssignableVolunteers()`, which still filters
   `WHERE role = 'volunteer'` (`src/server/db/repositories/profiles.ts:96-109`). If an
   admin manually assigns a `requested` pickup to a volunteer through this dialog, that
   user can never advance it afterward (`advancePickup`/`setProofPhoto`/`recordPing` all
   gate on `role === "driver"` and `pickup.volunteerId === userId`) — the pickup gets
   stuck. This dialog was seemingly never updated when the collector role moved from
   volunteer → driver. Worth a screenshot + a follow-up fix ticket.
5. Not a gap, but worth noting for the capture agent: `VolunteerClaimNote` (the quiet
   "you can't claim, browsing only" note shown to volunteers on the board and on a
   pickup's detail page) is an **intentional** dispatch-model-v2 replacement for the old
   claim button — not a leftover bug. Don't screenshot it as an error state.
