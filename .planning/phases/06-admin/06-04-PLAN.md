<objective>
Build the two highest-interactivity admin screens: `/admin/pickups` (ADM-01 list + filters
+ ADM-02 inline assign) and `/admin/users` (ADM-03 role change + deactivate/reactivate).
Both pages live under `src/app/admin/**` (D-02) and are thin server components that
re-check `requireRole(["admin"])` (D-01) and read
through the repos; the interactive bits (assign control, user-row controls, filter form)
are small client components calling the 06-03 actions. Reuse `PickupStatusPill`; build
everything from design tokens; reflect the self-guard in the UI (the admin's own row hides
its destructive controls) and surface the role-change claim-lag in copy.

Purpose: These deliver the staff-facing core of the phase — see all pickups, fix an
unassigned one, and manage who can do what. They are server-first per frontend-practices
§5 (data that doesn't need client interactivity stays on the server); only the mutating
controls are client components. The assign control must source volunteers from
`listAssignableVolunteers` (onboarded + non-deactivated, Open Q2) and the user controls
must mirror the action self-guard so an admin can't even attempt to lock themselves out.
Cross-ref RESEARCH "Recommended Project Structure", Pitfall P1 (claim-lag copy), the
CONTEXT "Specific Ideas" (functional + token-styled, desktop-first), and the existing
PickupForm/PickupCard for the form + row idioms.

Output: `pickups/page.tsx`, `users/page.tsx`, the four components, and the extended barrel.
</objective>

The admin page auth pattern to MIRROR (from src/app/admin/dashboard/page.tsx — already
exists; this is the existing, correct shape for an admin server component):
```typescript
import { redirect } from "next/navigation";
import { getSession, requireRole, AuthError } from "@/server/auth/session";
import { ROUTES } from "@/config/constants";
export default async function Page() {
  const session = await getSession();
  if (!session) redirect(ROUTES.signIn);
  try { await requireRole(["admin"]); }
  catch (e) { if (e instanceof AuthError) redirect(ROUTES.portalDashboard); throw e; }
  // ...read via repos, render...
}
export const dynamic = "force-dynamic"; // admin lists are always live
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> /admin/pickups, /admin/users | Server components re-check `requireRole(["admin"])`; a non-admin is redirected (middleware also 403s) |
| client controls -> admin actions | The assign/role/deactivate controls are conveniences; the server actions (06-03) are the real boundary and re-validate everything |
| searchParams -> listForAdmin | User-controlled filter params flow through `parseAdminFilters` (Zod) into the parameterized Drizzle query — no string concatenation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-14 | Elevation of Privilege | A non-admin reaching the admin pages via a direct RSC/URL hit | mitigate | Each page re-checks `requireRole(["admin"])` and redirects on AuthError (defense-in-depth over middleware) |
| T-06-15 | Denial of Service | Admin self-demote/self-deactivate attempted from the UI | mitigate | The own-row controls are disabled/hidden AND the action blocks it server-side (06-03) — double layer |
| T-06-16 | Tampering | Crafted filter query params (injection / type confusion) | mitigate | `parseAdminFilters` (Zod) validates + coerces; the repo uses parameterized `and()`/`eq()` — no SQL string building |
| T-06-17 | Information Disclosure | The assign dropdown exposing deactivated/non-onboarded volunteers | mitigate | Sourced from `listAssignableVolunteers` (onboarded + non-deactivated only); the action re-validates the chosen id against the same set |
</threat_model>
