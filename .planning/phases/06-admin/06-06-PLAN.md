This task is `auto` because the MCP tool performs it programmatically. If the MCP call

**Step A — confirm live state first** (Supabase guidance: `list_tables` before changes). If
`partners` already exists from a prior run, skip its `CREATE` and apply only the missing
RLS; if the profiles columns already exist, skip the `ALTER`.

**Step B — apply via the Supabase MCP `apply_migration` tool** (name it `0004_admin_partners`).
The app talks to Postgres as the `postgres` role (bypasses RLS); RLS here purely DENIES the
anon key / PostgREST (no policy = deny). Use the generated 0004 SQL as the source of truth
(match it exactly); the statements below are the expected shape:

```sql
-- enum + table (match the Drizzle 0004 migration; idempotent guards for re-runs)
do $$ begin
  create type partner_type as enum ('restaurant','hall','event_planner','family','other');
exception when duplicate_object then null; end $$;

create table if not exists public.partners (
  id            text primary key,
  name          text not null,
  type          partner_type not null,
  contact_name  text,
  contact_phone text,
  contact_email text,
  address       text,
  city          text not null default 'Ahmedabad',
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

alter table public.profiles add column if not exists partner_id text references public.partners(id);
alter table public.profiles add column if not exists deactivated_at timestamptz;

-- RLS: enable on partners with NO policy -> anon/PostgREST denied by default.
-- The server (postgres role) bypasses RLS, so the app keeps working. The two new
-- profiles columns inherit profiles' existing RLS (A4).
alter table public.partners enable row level security;
```

**Step C — verify with the Supabase MCP** (Supabase guidance: `get_advisors` after schema

- `get_advisors` (security) reports no "RLS disabled" / policy-gap warning for `partners`,
  and confirms `profiles` RLS is intact (A4).

## STRIDE Threat Register

| T-06-24 | Elevation of Privilege | A non-admin reaching the deployed admin pages / export route | mitigate | Server-side `requireRole(["admin"])` on every page + the export route; verified in prod (Task 3) |
