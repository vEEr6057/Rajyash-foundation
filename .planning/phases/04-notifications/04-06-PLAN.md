returns an auth error, THEN it converts to a checkpoint (the human authenticates the
MCP/Supabase connection and re-runs).

**Step A — confirm live state first** (Supabase guidance: `list_tables` before changes).
If a table already exists from a prior run, skip its `CREATE TABLE` and apply only the
missing RLS.

**Step B — apply via the Supabase MCP `apply_migration` tool** (name it
`0003_notifications`). The app talks to Postgres as the `postgres` role (bypasses RLS);
RLS here is purely to DENY the anon key / PostgREST (no policy = deny). These tables are
NOT in the realtime publication (in-app reads go through server actions, not Realtime).

```sql
-- Tables (match the Drizzle 0003 migration; idempotent guards for re-runs)
create table if not exists public.notifications (
  id          text primary key,
  user_id     text not null references public.profiles(id),
  type        text not null,
  title       text not null,
  body        text not null,
  data        jsonb,
  pickup_id   text references public.pickups(id) on delete cascade,
  read_at     timestamptz,
  created_at  timestamptz not null default now()
);
create index if not exists notifications_user_idx on public.notifications (user_id);
create index if not exists notifications_user_unread_idx on public.notifications (user_id, read_at);

create table if not exists public.push_subscriptions (
  id          text primary key,
  user_id     text not null references public.profiles(id),
  endpoint    text not null unique,
  p256dh      text not null,
  auth        text not null,
  created_at  timestamptz not null default now()
);
create index if not exists push_subscriptions_user_idx on public.push_subscriptions (user_id);

create table if not exists public.notification_deliveries (
  id            text primary key,
  event_id      text not null,
  recipient_id  text not null,
  channel       text not null,
  created_at    timestamptz not null default now(),
  constraint notif_delivery_unique unique (event_id, recipient_id, channel)
);

-- RLS: enable on all three; add NO policy -> anon/PostgREST is denied by default.
-- The server (postgres role) bypasses RLS, so the app keeps working.
alter table public.notifications enable row level security;
alter table public.push_subscriptions enable row level security;
alter table public.notification_deliveries enable row level security;
```

**Step C — verify with the Supabase MCP** (Supabase guidance: `get_advisors` after

- RLS is enabled on all three; no anon-readable policy exists.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anon key / PostgREST -> notification tables | Public anon key must NOT read/write these; RLS deny-by-default (no policy) is the gate |
| Inngest Cloud -> /api/inngest (prod) | Authenticated by the signing-key signature; INNGEST_SIGNING_KEY is a Cloudflare secret |
| VAPID private key / Resend key -> Workers | Server-only Cloudflare secrets; never inlined into the client bundle |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-23 | Information Disclosure | Notification tables readable via the anon key before RLS | mitigate | Task 1 enables RLS with no anon policy on all three; `get_advisors` confirms no gap |
| T-04-24 | Spoofing | Forged production calls to /api/inngest | mitigate | Signing-key signature verified by `serve()`; key set as a Cloudflare secret in Task 2 |
| T-04-25 | Information Disclosure | VAPID private / Resend key leaked via a build var | mitigate | Only `NEXT_PUBLIC_VAPID_PUBLIC_KEY` is a plain build var; the private key + Resend key are encrypted Cloudflare secrets |
| T-04-26 | Repudiation | Replayed event double-sends in production | mitigate | Verified live in Task 3 step 10–11 (replay -> no dup; the group-by returns zero); enforced by the UNIQUE constraint + Inngest idempotency |
| T-04-27 | Privacy | Email reaches an unintended recipient | accept/mitigate | Sandbox only delivers to the account owner; `from=onboarding@resend.dev` until a domain is verified (deferred); recipient is the server-resolved donor, not client input |
</threat_model>
