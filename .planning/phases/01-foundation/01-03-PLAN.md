The challenge: `createEnv` validates at module evaluation time, so we cannot import `env`
from `src/config/env.ts` in tests that want to test invalid env. The test must use
`SKIP_ENV_VALIDATION` to isolate env from the module cache.

```typescript
// src/config/env.test.ts
// Tests env validation schema behavior — AUTH-06 / D-06.
// We test the SCHEMA directly (createEnv schema objects) rather than the exported singleton,
// since the singleton validates at module load and is cached by Node's module system.

import { z } from 'zod';

// The server schema from env.ts — extracted for testability
const serverSchema = {
  DATABASE_URL: z.string().url(),
  DIRECT_URL: z.string().url(),
  CLERK_SECRET_KEY: z.string().min(1),
  NODE_ENV: z.enum(['development', 'test', 'production']).default('development'),
};

const clientSchema = {
  NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY: z.string().min(1),
};

const fullSchema = z.object({ ...serverSchema, ...clientSchema });

const validEnv = {
  DATABASE_URL: 'postgres://user:pass@aws-us-east-1.pooler.supabase.com:6543/postgres',
  DIRECT_URL: 'postgresql://user:pass@db.project.supabase.co:5432/postgres',
  CLERK_SECRET_KEY: 'sk_test_abc123',
  NODE_ENV: 'development' as const,
  NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY: 'pk_test_abc123',
};

describe('env validation schema', () => {
  it('validates successfully when all required vars are present', () => {
    const result = fullSchema.safeParse(validEnv);
    expect(result.success).toBe(true);
  });

  it('rejects when DATABASE_URL is missing', () => {
    const { DATABASE_URL: _, ...rest } = validEnv;
    const result = fullSchema.safeParse(rest);
    expect(result.success).toBe(false);
    if (!result.success) {
      expect(result.error.issues.some(i => i.path.includes('DATABASE_URL'))).toBe(true);
    }
  });

  it('rejects when DATABASE_URL is not a valid URL', () => {
    const result = fullSchema.safeParse({ ...validEnv, DATABASE_URL: 'not-a-url' });
    expect(result.success).toBe(false);
    if (!result.success) {
      expect(result.error.issues.some(i => i.path.includes('DATABASE_URL'))).toBe(true);
    }
  });

  it('rejects when CLERK_SECRET_KEY is empty string', () => {
    const result = fullSchema.safeParse({ ...validEnv, CLERK_SECRET_KEY: '' });
    expect(result.success).toBe(false);
    if (!result.success) {
      expect(result.error.issues.some(i => i.path.includes('CLERK_SECRET_KEY'))).toBe(true);
    }
  });

  it('rejects when NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY is missing', () => {
    const { NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY: _, ...rest } = validEnv;
    const result = fullSchema.safeParse(rest);
    expect(result.success).toBe(false);
    if (!result.success) {
      expect(result.error.issues.some(i =>
        i.path.includes('NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY')
      )).toBe(true);
    }
  });

  it('defaults NODE_ENV to "development" when not provided', () => {
    const { NODE_ENV: _, ...rest } = validEnv;
    const result = fullSchema.safeParse(rest);
    expect(result.success).toBe(true);
    if (result.success) {
      expect(result.data.NODE_ENV).toBe('development');
    }
  });
});
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server env vars → client bundle | @t3-oss/env-nextjs enforces the server/client split |
| process.env → runtime code | Zod validates type and format at boot, not just presence |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | Server env vars in client bundle | mitigate | @t3-oss/env-nextjs server/client split — CLERK_SECRET_KEY and DATABASE_URL are `server:` only; they never appear in NEXT_PUBLIC_* |
| T-03-02 | Information Disclosure | .env.local placeholder values | accept | Placeholder values are fake; .env.local is gitignored; README warns developers to replace before testing real auth |
| T-03-03 | Denial of Service | Missing env var at boot | mitigate | Desired behavior: app refuses to start. createEnv throws with a clear Zod error listing which vars are missing |
| T-03-04 | Elevation of Privilege | Role values read from request body | mitigate | Role must only be read from sessionClaims?.metadata?.role (typed via CustomJwtSessionClaims). The ROLES constant makes the allowed values explicit; nothing reads role from req.body |
</threat_model>
