Pattern 3 — DB client (Cloudflare Workers constraint):
```typescript
// src/server/db/client.ts
import 'server-only';
import { drizzle } from 'drizzle-orm/postgres-js';
import postgres from 'postgres';
import { cache } from 'react';
import * as schema from './schema';

// POOL_URL = Supabase Supavisor transaction-mode URL (port 6543)
// prepare: false is MANDATORY — transaction mode does not support prepared statements.
// cache() scopes to one request — DO NOT create a module-level global client.
export const getDb = cache(() => {
  const client = postgres(process.env.DATABASE_URL!, { prepare: false });
  return drizzle({ client, schema });
});
```


- ❌ `const db = drizzle(postgres(url))` at module level (global client — breaks Workers)
- ❌ `postgres(url, { prepare: true })` or default (transaction pooler incompatibility)
- ❌ `export const runtime = "edge"` in any route file (CF adapter incompatibility)

Use `env.DATABASE_URL` from src/config/env.ts instead of `process.env.DATABASE_URL!` directly,
so the Zod-validated env is the single source of truth:

```typescript
// src/server/db/client.ts
import 'server-only';
import { drizzle } from 'drizzle-orm/postgres-js';
import postgres from 'postgres';
import { cache } from 'react';
import * as schema from './schema';
import { env } from '@/config/env';

// Per-request DB client. Must use cache() — Cloudflare Workers cannot reuse TCP connections
// across requests. A module-level client would work on request 1, then hang/fail on request 2.
//
// prepare: false is MANDATORY for Supabase Supavisor transaction mode (port 6543).
// Prepared statements are not supported in transaction-pooler mode. The default (prepare: true)
// causes "prepared statement does not exist" errors silently on the pooler.
export const getDb = cache(() => {
  const client = postgres(env.DATABASE_URL, { prepare: false });
  return drizzle({ client, schema });
});
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase pooler → DB client | TCP connection per request via postgres-js; no persistent connection |
| Server action → DB layer | Only server-side code (server-only guard) can call getDb() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | DATABASE_URL in server bundle | mitigate | `server-only` import guard in client.ts; env.ts splits DATABASE_URL into server-only. No client component can import getDb() — build fails with "You're importing a component that needs server-only" |
| T-04-02 | Tampering | Raw DB row types in client components | mitigate | Types exported from auth.types.ts (InferSelectModel); components import from feature barrel, not from schema directly. Mapper layer prevents raw shapes reaching JSX |
| T-04-03 | Elevation of Privilege | Role value in profiles table | mitigate | Role is written ONLY by server actions that first call requireRole(). No user-controlled input reaches the role column directly. Clerk publicMetadata is the auth source of truth; DB is a read-optimized copy |
| T-04-04 | Denial of Service | Global postgres client (Pitfall 1) | mitigate | Explicitly using cache() wrapper. Any future PR that removes cache() will fail on request 2 in Cloudflare Workers — the pattern is self-enforcing at runtime |
</threat_model>
